# Agent 指南

## 1. 定位

- 本文件是项目级长期纲领，适用于后续 agent、开发者和维护者协作。
- 默认目标：把项目做成可公开展示、可复用、可维护、可部署、可演示的作品。
- 当前明确需求优先；若涉及安全、数据丢失或生产风险，必须说明风险并请求确认。
- 本文件只保留精简规则、仓库地图、文档入口和常用命令；细则放入 `docs/`。

## 2. 快速规则

- 先读代码、测试、README 和相关 `docs/` 文档，再改代码。
- 生产问题先取证再修复：查浏览器 Console、后端日志、Nginx 日志、容器状态、数据库状态和时间线。
- 改动保持聚焦，不把无关重构、依赖升级、样式调整和功能修复混在一起。
- 生产问题修复必须补回归测试，或说明不能补的原因。
- 出现问题并修复后，要主动复盘为何没被提前发现，检查 `docs/` 细则、测试、脚本或配置是否缺失，并给出改进方案让用户确认。
- 每次完成验证或排障收尾前，都要主动检查是否存在“下次还会踩”的流程坑、命令误用、环境前置条件或文档缺口；若存在，优先沉淀为文档、脚本提示、测试、预检或配置，并在最终回复中说明防复发措施。
- 发现现有纲领、细则、依赖、测试策略或部署方式可能过时时，要说明新方案、收益、风险和迁移成本，作为建议交给用户确认，不擅自大改。
- 不硬编码或提交本机路径、私有域名、密钥、Token、Cookie、数据库密码或个人机器配置。
- 用户提示、日志、反馈和审计不得暴露底层异常、堆栈、完整 Cookie、密钥或不必要的敏感信息。
- 所有 Markdown 文档使用简体中文；必要英文术语可以保留，但说明文字优先中文。
- 编辑中文文档或源码前后，按 UTF-8 读取确认内容，不把终端乱码误判为文件损坏。
- 发现重复流程、常见坑或约束缺失，要沉淀为文档、脚本、Skill、测试或配置。
- 新增 cron、后台任务、日志、缓存或备份时，必须说明磁盘增长、保留策略、失败可见性和人工检查方式。

## 3. 文档入口

- 测试、回归、前端发布后 smoke test：读 `docs/testing.md`。
- 部署、服务器更新、HTTPS、健康检查：读 `docs/deployment.md`。
- 生产问题排障、日志取证、故障复盘：读 `docs/troubleshooting.md`。
- 密钥、权限、Cookie、CORS、日志脱敏和生产安全：读 `docs/security.md`。
- 备份、恢复、数据丢失和自动备份：读 `docs/backup-restore.md`。
- 架构、模块边界、数据流：读 `docs/architecture.md`。
- 重要技术选择、长期取舍、升级建议：读 `docs/decisions.md`。
- 生产就绪检查：读 `docs/PRODUCTION_READINESS_CHECKLIST.md`。
- 如果新增长流程，优先写入 `docs/`；重复 3 次以上的专项流程再考虑沉淀为 Skill。
- 如果发现文档、脚本或配置已落后于更好的工程实践，先提出更新建议和取舍，再由用户确认是否沉淀。

## 4. 仓库地图

- `backend/`：FastAPI/RAG 后端、业务逻辑、数据库访问、外部模型服务适配。
- `frontend/`：Vue 前端、页面、组件、状态管理和前端测试。
- `docs/`：架构、部署、测试、运维和决策文档。
- `scripts/`：预检、评测、数据处理和运维辅助脚本。
- `compose*.yml`、`Dockerfile`、`nginx*.conf`：容器、反向代理和部署配置。
- `knowledge_files/`、`rag_eval/`：知识库文件和评测数据。
- `.github/workflows/`：CI 和手动 deploy workflow。

## 5. 常用命令

```bash
cd frontend
npm.cmd test
npm.cmd run build
```

```powershell
.\.venv\Scripts\python.exe run_tests.py
```

```bash
docker compose --env-file .env.prod -f compose.prod.yml pull
docker compose --env-file .env.prod -f compose.prod.yml up -d
docker compose --env-file .env.prod -f compose.prod.yml ps
docker logs ai-agent-backend --tail 100
docker logs ai-agent-nginx --tail 100
docker inspect ai-agent-backend --format '{{json .State.Health}}'
docker inspect ai-agent-nginx --format '{{json .State.Health}}'
```

- 以当前项目的 `package.json`、README、测试脚本和 CI 配置为准。
- PowerShell 中若 `npm` 被执行策略拦截，使用 `npm.cmd`。
- 服务器日志时间可能是 UTC。用户描述本地时间时，要换算并明确时间窗口。
- Nginx `499` 常见于客户端断开；`502/504` 常见于代理到后端失败或超时。

## 6. 验证原则

- 改前端组件、状态、格式化、Vue `required` props、父子组件绑定、聊天渲染、来源展示或反馈按钮：按 `docs/testing.md` 的前端清单验证。
- 改后端路由、鉴权、schema：跑对应后端测试，至少覆盖成功、失败、未授权。
- 改数据库访问或结构：跑数据库相关测试，并说明迁移、回滚和历史数据兼容。
- 改 RAG、检索、提示词、文档解析、权限过滤：跑相关单测和评测，或说明评测缺口。
- 改 Docker、Nginx、部署脚本、环境变量：按部署文档做预检，检查健康检查、日志、数据卷和回滚路径。
- 只改文档：按 UTF-8 读回确认中文正常，检查命令是否跨平台或注明平台限制。
- 验证通过后不要只报告“测试通过”；还要说明是否发现并处理了防复发事项，或明确本次没有需要新增沉淀的流程缺口。

## 7. 工程原则

- 方案取舍默认排序：安全可靠、易部署、易运维、易测试、易理解、性能优化、视觉或技术炫技。
- 命名必须反映真实行为，避免历史遗留名称误导维护。
- 模块边界要清晰：业务逻辑、接口层、存储层、外部服务适配层和 UI 状态不要无故混在一起。
- 配置通过环境变量、配置文件或部署参数注入；秘密值只能通过安全配置注入。
- 错误处理区分用户可理解提示和开发者诊断信息，不把内部异常原样返回给用户。
- 外部输入必须校验；外部服务调用必须考虑超时、重试、限流、降级和错误记录。
- 权限检查放在服务端，前端显示控制不能替代后端授权。
- 前端失败但后端可能已完成时，应优先恢复状态，例如重新拉取会话、订单、任务或上传结果。

## 8. 本项目专项

- 当前项目优先服务中小企业 RAG 部署场景，关注安全基线、部署简洁性、运维可控性、成本可预测性和知识库效果稳定性。
- 回答质量问题要区分“模型没有答好”和“知识库证据不足”，同时查看召回来源、分数、部门权限、文档版本和最终提示词。
- 回答应保留来源信息。前端、反馈、审计和评测链路都要能追踪答案使用了哪些知识库片段。
- RAG 变更需要评测兜底；模型相关能力要关注 token 成本、并发限制和失败降级。
- 对中小企业默认建议偏保守：同源部署、HTTPS、强口令、最小暴露面、可观测性、备份恢复和清晰运维手册优先。
