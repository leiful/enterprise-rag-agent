# 架构说明

本文档描述项目的核心架构、数据边界和 RAG 处理链路，帮助维护者和部署者快速理解系统设计。

## 设计目标

本项目按中小企业 RAG 部署场景优化，重点关注：

- 部署简洁
- 成本可控
- 审计可追踪
- 运维可观测
- 效果稳定

## 系统组成

```text
frontend (Vue 3)
  -> backend (FastAPI)
    -> PostgreSQL
    -> pgvector (in PostgreSQL)
    -> Model APIs
    -> knowledge_files
```

### 前端

前端提供浏览器控制台，主要覆盖：

- 聊天界面
- 会话管理
- 知识库管理
- 用户与管理员界面
- 运维与 RAG 评测入口
- token 监控

前端通过 `VITE_API_BASE` 决定 API 基地址。生产环境推荐同源部署，减少跨域与 Cookie 配置复杂度。

### 后端

后端基于 FastAPI，承担以下职责：

- 登录认证与会话管理
- 聊天与流式响应
- 知识文档解析、分块、索引和同步
- 混合检索、重排与答案生成
- 用户、反馈、审计与模型用量记录
- 运维状态和评测接口

### PostgreSQL

PostgreSQL 存储结构化业务数据，包括：

- 用户和会话
- 对话与消息
- 知识文档元数据
- chunk 元数据
- BM25 相关索引数据
- 知识源与索引任务
- 反馈、审计和模型用量

### pgvector (向量索引)

向量索引和检索所需的向量数据直接存储在 PostgreSQL 中（通过 pgvector 扩展），与业务数据共享同一数据库，无需独立的向量存储服务。

## RAG 处理链路

典型链路如下：

1. 上传文件或同步本地知识源
2. 解析文档内容
3. 执行标题感知分块，可选语义分块
4. 生成 embedding 并写入 pgvector
5. 写入 PostgreSQL 元数据和 BM25 数据
6. 用户发起问题
7. 执行查询改写、多查询扩展和混合检索
8. 可选重排候选片段
9. 结合检索证据生成回答
10. 记录反馈、审计和模型用量

## 检索策略

当前检索能力以“质量优先、成本可控”为目标，适合中小企业常见知识库规模：

- 向量检索
- BM25 检索
- 混合召回
- 查询改写
- 多查询扩展
- 重排
- 查询覆盖率过滤

这些策略的开关和阈值可以通过环境变量调整。

## 安全与访问边界

### 认证

- 浏览器通过 `/login` 登录
- 后端将 session 存储在 PostgreSQL
- 通过 `HttpOnly` Cookie 维护会话

### 访问控制

- 受保护路由要求携带有效 session
- 知识访问可结合部门信息进行过滤
- 管理动作会记录审计事件

### 密钥边界

- 模型 API key 和 embedding API key 只应存在于后端运行环境
- 前端 `VITE_` 变量会被打包到浏览器，不应用于存放任何密钥

## 部署边界

推荐的部署边界是：

- 前端只暴露构建产物 `frontend/dist/`
- 后端通过服务进程或容器运行
- PostgreSQL 数据和向量索引保持私有
- `knowledge_files/`、日志目录和 `.env` 不公开暴露

这套边界兼顾了中小企业常见的单机部署需求和后续扩展空间。

## 可观测性

当前项目已具备若干适合运维排查的入口：

- `/health`
- `/admin/rag/status`
- `/admin/rag/eval`
- `/admin/audit`
- `/admin/knowledge-audit`

同时还保留模型用量和部分运维可视化页面，便于观察调用量、评测结果和知识库状态。

## 扩展方向

围绕中小企业部署场景，后续适合优先演进的方向包括：

- 对象存储知识源
- 企业文档系统连接器
- 更细粒度的权限模型
- 更完整的告警与备份恢复机制
- 更标准的容器化和多环境部署模板
