# 决策记录

## 1. 使用范围

以下情况需要记录决策：

- 更换主要技术栈、框架、数据库、模型服务或部署方式。
- 引入新的长期依赖、CI/CD 策略、测试框架或运维工具。
- 改变安全边界、权限模型、数据存储位置或日志策略。
- 对 RAG 检索、重排、提示词、评测门禁做影响长期效果的调整。
- 接受明显技术债或推迟重要改造。

小修小补不需要记录；能在代码注释或提交信息里说清楚的，不必写成长决策。

## 2. 决策原则

- 记录背景和取舍，不只记录结论。
- 优先选择安全可靠、易部署、易运维、易测试、易理解的方案。
- 新技术方案要说明收益、风险、迁移成本和回滚方式。
- 重要决策应让用户确认后再落地。
- 决策不是永久正确；当技术进步或项目阶段变化时，应主动建议复审。

## 3. 决策模板

```md
## YYYY-MM-DD 决策标题

### 背景

为什么需要做这个决策。

### 选择

最终选择的方案。

### 替代方案

- 方案 A：优点、缺点。
- 方案 B：优点、缺点。

### 取舍

为什么当前阶段选择它，放弃了什么。

### 风险

可能出问题的地方。

### 验证

如何证明该决策有效。

### 回看点

什么时候应该重新评估这个决策。
```

## 4. 已有重要决策

### 2026-06-24 AGENTS 作为纲领，细则放入 docs

#### 背景

项目需要长期沉淀 agent 协作规则，但 `AGENTS.md` 如果承载所有细节，会越来越难读。

#### 选择

`AGENTS.md` 只保留纲领、入口和关键规则；测试、部署、排障、安全等细则放入 `docs/`。

#### 取舍

这样能保持入口简洁，也让细则可以独立演进。代价是 agent 必须根据任务主动阅读对应文档。

#### 回看点

如果某类流程重复 3 次以上，考虑从 `docs/` 抽成 Skill 或脚本。

### 2026-06-24 先保留手动部署，不立即做完整 CD

#### 背景

当前项目已有 CI 和手动镜像发布，但服务器更新仍由人工执行。

#### 选择

短期保留手动触发 deploy 和服务器手动 `docker compose pull/up -d`。

#### 取舍

保留人工发布闸门，降低生产误操作风险。代价是发布步骤仍需人工执行。

#### 回看点

当服务器手动更新成为频繁负担，或发布检查稳定后，再考虑手动触发 CD。
